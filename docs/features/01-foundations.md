# Sprint 01 — Foundations

Set up the bare project so we have a deployable Next.js app connected to Supabase (local + cloud), with Drizzle ORM ready, TypeScript strict, Tailwind + shadcn/ui, next-intl wired for EN/FR, and the Léon & George org seeded. By the end of this sprint, `supabase start` boots the full Supabase stack locally via Docker, `npm run dev` shows a "hitempo" page reading from the local Postgres, and the same app is deployed on Vercel against a Supabase Cloud project.

## Why this layout

Repo root holds the Next.js project (Vercel convention, zero rootDirectory config). A sibling `supabase/` folder holds the Supabase CLI workspace — a managed docker-compose under the hood: Postgres + Auth + Storage + Realtime + Studio.

Rationale:

- **Iso-prod local stack.** Supabase CLI runs the same services we use in production, including `auth.uid()` and RLS — both critical to hitempo's multi-tenant guarantees. We refuse the "Postgres local + cloud Auth" trap that breaks RLS testing.
- **No home-grown docker-compose to maintain.** Supabase ships and maintains the compose file. `supabase start`, `supabase stop`, `supabase db reset` are the only commands the team needs to learn.
- **Drizzle stays the source of truth for the schema.** We generate SQL with `drizzle-kit generate`, copy the resulting file into `supabase/migrations/`, and `supabase db reset` replays it. Two complementary tools, one source of truth.
- **Vercel-friendly.** The repo root is a vanilla Next.js project — Vercel detects, builds, deploys it. The `supabase/` folder is irrelevant to the build and ignored by `.vercelignore` if needed.

### Step 8 — Drizzle config and client

Create `drizzle.config.ts` at the repo root:

```typescript
import type { Config } from "drizzle-kit";
import "dotenv/config";

export default {
  schema: "./db/schema.ts",
  out: "./db/.drizzle-out",       // staging area; sync script copies into supabase/migrations/
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.SUPABASE_POSTGRES_DIRECT_URL!,
  },
  verbose: true,
  strict: true,
} satisfies Config;
```

Why an intermediate `db/.drizzle-out` instead of pointing directly at `supabase/migrations`? Drizzle-kit owns its output directory (it writes meta files, tracks state). Supabase CLI also owns `supabase/migrations` (timestamped filenames, journal). We let each tool own its folder, and we copy the generated SQL across with a small script (step 10). This keeps both tools' assumptions intact.

Create `db/client.ts`:

```typescript
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "./schema";

if (!process.env.SUPABASE_POSTGRES_URL) {
  throw new Error("SUPABASE_POSTGRES_URL is required");
}

const client = postgres(process.env.SUPABASE_POSTGRES_URL, {
  prepare: false, // required for Supabase transaction pool
});

export const db = drizzle(client, { schema });

// Admin client (RLS bypassed) — server-only, never imported from client code
let _adminDb: ReturnType<typeof drizzle> | null = null;
export function getAdminDb() {
  if (!_adminDb) {
    if (!process.env.SUPABASE_POSTGRES_DIRECT_URL) {
      throw new Error("SUPABASE_POSTGRES_DIRECT_URL required");
    }
    const adminClient = postgres(process.env.SUPABASE_POSTGRES_DIRECT_URL, {
      prepare: false,
    });
    _adminDb = drizzle(adminClient, { schema });
  }
  return _adminDb;
}
```

### Step 9 — First schema (organizations + organization_members)

Create `db/schema.ts`:

```typescript
import { pgTable, uuid, text, jsonb, timestamp, pgEnum, uniqueIndex, index } from "drizzle-orm/pg-core";
import { sql } from "drizzle-orm";

export const organizationPlan = pgEnum("organization_plan", [
  "trial", "starter", "pro", "business",
]);

export const memberRole = pgEnum("member_role", [
  "owner", "admin", "commercial", "viewer",
]);

export const organizations = pgTable("organizations", {
  id: uuid("id").primaryKey().defaultRandom(),
  slug: text("slug").notNull().unique(),
  name: text("name").notNull(),
  plan: organizationPlan("plan").notNull().default("trial"),
  defaultLocale: text("default_locale").notNull().default("fr"),
  supportedLocales: text("supported_locales").array().notNull().default(sql`ARRAY['fr', 'en']`),
  brandBrief: jsonb("brand_brief").default({}),
  settings: jsonb("settings").default({}),
  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).notNull().defaultNow(),
});

export const organizationMembers = pgTable("organization_members", {
  id: uuid("id").primaryKey().defaultRandom(),
  organizationId: uuid("organization_id").notNull().references(() => organizations.id, { onDelete: "cascade" }),
  userId: uuid("user_id").notNull(),
  role: memberRole("role").notNull().default("commercial"),
  preferredLocale: text("preferred_locale").notNull().default("fr"),
  timezone: text("timezone").notNull().default("Europe/Paris"),
  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).notNull().defaultNow(),
}, (t) => ({
  uniqMember: uniqueIndex("uniq_org_user").on(t.organizationId, t.userId),
  byUser: index("idx_org_members_user").on(t.userId),
}));
```

```bash

```

You'll get a file under `db/.drizzle-out/0000_init_organizations.sql`. Open it, append the RLS bootstrap at the end (RLS policies for business tables come in sprint 03; here we set up the helper and the minimal self-read policies):

```sql
-- Enable RLS (business policies added in sprint 03)
ALTER TABLE organizations ENABLE ROW LEVEL SECURITY;
ALTER TABLE organization_members ENABLE ROW LEVEL SECURITY;

-- Helper function for RLS (used by future policies on all business tables)
CREATE OR REPLACE FUNCTION public.user_organization_ids() RETURNS SETOF uuid
LANGUAGE sql STABLE SECURITY DEFINER SET search_path = public AS $$
  SELECT organization_id FROM organization_members WHERE user_id = auth.uid()
$$;

-- Allow authenticated users to read their own memberships
CREATE POLICY "users_read_own_memberships" ON organization_members FOR SELECT
  USING (user_id = auth.uid());

-- Allow authenticated users to read orgs they belong to
CREATE POLICY "users_read_own_orgs" ON organizations FOR SELECT
  USING (id IN (SELECT public.user_organization_ids()));
```

Create `db/sync-migrations.ts`:

```typescript
import { readdirSync, copyFileSync, existsSync, mkdirSync, readFileSync } from "node:fs";
import { join, resolve } from "node:path";

const drizzleOut = resolve(__dirname, ".drizzle-out");
const supabaseMigrations = resolve(__dirname, "..", "supabase", "migrations");

if (!existsSync(drizzleOut)) {
  console.error("No drizzle output. Run `npm run db:generate` first.");
  process.exit(1);
}
mkdirSync(supabaseMigrations, { recursive: true });

const existing = new Set(readdirSync(supabaseMigrations));
const drizzleFiles = readdirSync(drizzleOut).filter((f) => f.endsWith(".sql"));

let copied = 0;
for (const file of drizzleFiles) {
  // Drizzle names: 0000_init_xxx.sql, 0001_yyy.sql ...
  // Supabase names: YYYYMMDDHHMMSS_xxx.sql
  // Keep the Drizzle "name" part; prefix with a fresh timestamp on first sync.
  const drizzleName = file.replace(/^\d+_/, "");
  const alreadyCopied = [...existing].some((e) => e.endsWith(`_${drizzleName}`));
  if (alreadyCopied) continue;

  const ts = new Date().toISOString().replace(/[-T:Z.]/g, "").slice(0, 14);
  const targetName = `${ts}_${drizzleName}`;
  const sql = readFileSync(join(drizzleOut, file), "utf8");
  copyFileSync(join(drizzleOut, file), join(supabaseMigrations, targetName));
  console.log(`copied -> supabase/migrations/${targetName} (${sql.length} bytes)`);
  copied++;
}

if (!copied) console.log("nothing to sync");
```

(For sprint 01, since we have a single migration, you can also just rename and copy by hand — the script earns its keep starting sprint 02.)

Then:

```bash
npm run db:sync
npm run db:reset    # drops the local DB and replays everything under supabase/migrations/
```

Verify in Supabase Studio at `http://127.0.0.1:54323` that both tables exist and RLS is enabled.

## Implementation notes

Executed on 2026-05-26 on Ludovic's macOS (Apple Silicon).

**Environment setup gotchas**

- macOS Command Line Tools were too outdated for `brew install supabase/tap/supabase` to compile. Bypassed by downloading the prebuilt binary from GitHub releases (`v2.101.0`, `supabase_2.101.0_darwin_arm64.tar.gz`) into `~/.local/share/supabase/` and symlinking `supabase` and its `supabase-go` companion into `~/.local/bin/`. **Both binaries must live in the same directory** — the wrapper looks for `supabase-go` alongside `supabase` on PATH. If brew gets fixed later, the brew install can replace the manual one without changing anything project-side.
- Node was at v16 (system default). Initially landed on v20.20.2, then switched to **v24.16.0** after Ludovic pointed out that Node 20 had just exited Active LTS (April 2026). Repo-level `.nvmrc` pinned to `24`, `engines.node >= 24` added to `package.json`, `@types/node` bumped to `^24`. Lint, build, and DB seed re-validated on Node 24 cleanly. **Rule for next sprints:** pick the most recent Active LTS, not the minimum required by the framework.

**Stack deviations from the brief**

- `create-next-app` shipped **Next.js 16.2.6** (the brief mentioned 15). The scaffold also added an `AGENTS.md` with the warning *"This is NOT the Next.js you know"* — assume some APIs and conventions may have shifted; read `node_modules/next/dist/docs/` when in doubt. Build, lint, dev all pass with the next-intl wiring as written, so no immediate breakage.
- `npx create-next-app .` does not accept `--turbopack` as a flag any more (it is the default in v16). Removed from the command. `next dev` is the default script (no `--turbopack` switch needed).
- `shadcn@latest init` no longer accepts `--base-color` / `--style` / `--css-variables=yes` as in older versions. The new CLI options are `--template`, `--base`, `--monorepo`, `--defaults`, etc. Used `--defaults --yes --no-monorepo` which selected the Next.js + base preset. Components live under `components/ui/`, the `cn` helper under `lib/utils.ts`. Slate base color and CSS variables came as part of the defaults.
- Supabase CLI v2.101.0 (Postgres 17) exposes the **new key format**: `Publishable` (replaces `anon`) and `Secret` (replaces `service_role`). The `.env.local` was filled with these new-format keys — they still work as `NEXT_PUBLIC_SUPABASE_ANON_KEY` / `SUPABASE_SERVICE_ROLE_KEY` in the app code. When sprint 02 wires `@supabase/ssr`, double-check the lib handles the new format (it should — same JWT shape).
- `supabase start` started the core services (db, auth, kong, storage, rest, realtime, analytics, vector, inbucket, edge_runtime, pg_meta) but left `imgproxy`, `studio`, `pooler` stopped.
