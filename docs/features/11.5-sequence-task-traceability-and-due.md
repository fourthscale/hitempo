# Sprint 11.5 — Tâches issues d'une séquence : traçabilité, exécution, échéance

Polish post-Phase A pour que les tâches générées par l'engine soient :
- **lisibles** : on sait d'où elles viennent et où on en est dans la séquence
- **planifiées intelligemment** : moment d'exécution + échéance distincts,
  TZ contact + sale respectées, anti-conflit + quotas, plage de travail
- **modifiables** : le sale peut replanifier, l'échéance peut être absente

## Prérequis

- Sprint 11 Phase A live
- Engine Inngest opérationnel
- `tasks.scheduledFor` + `tasks.dueAt` déjà sur le schéma (sous-utilisés)

## Modèle proposé

### Sémantique dates tâche

```
scheduledFor  → « fais cette tâche à ce moment précis »
dueAt         → « après ça, c'est en retard »
dueAtAllDay   → true = afficher dueAt sans heure (échéance "le jour J")
priority      → low | medium | high | urgent (existe déjà)
```

Combinaisons couvertes :
| Cas usage | scheduledFor | dueAt | dueAtAllDay | priority |
|---|---|---|---|---|
| « Appelle Jean lundi 10h30, en retard mardi » | 2027-01-25 10:30 | 2027-01-26 00:00 | false | medium |
| « Prévois d'appeler Jean lundi, en retard mardi » | 2027-01-25 09:00 | 2027-01-26 | **true** | medium |
| « Rappelle Jean, priorité haute » | null | null | — | high |
| « Tâche urgente sans date » | null | null | — | urgent |

### Configuration sur la step (`action_config.scheduling`)

```ts
type TaskScheduling = {
  // Heures (TZ contact)
  preferredHour?: number;        // défaut 9
  businessHours?: { start; end }; // défaut {9, 18}
  allowedWeekdays?: number[];    // défaut [1..5]
  estimatedDurationMinutes?: number; // défaut 5 email / 15 call

  // scheduledFor
  setScheduledFor?: boolean;     // défaut true
  scheduledOffsetBusinessDays?: number; // défaut 0 = prochain jour ouvré

  // dueAt
  setDueAt?: boolean;            // défaut false
  dueOffsetBusinessDays?: number; // depuis scheduledFor, défaut 1 si setDueAt
  dueAtAllDay?: boolean;         // défaut true (échéance = fin de journée)
};
```

### Work pattern sale (`organization_members.work_pattern`)

```jsonb
{
  "weekday_windows": [
    {"start": "09:00", "end": "12:00"},
    {"start": "14:00", "end": "17:00"}
  ],
  "saturday": [],
  "sunday": []
}
```

V1 : un pattern unique pour tous les jours ouvrés. V2 : `byType` pour
plages par type de tâche (`call`, `visit`, `email`, …).

### Algorithme `computeTaskDueAt` (pur)

```ts
function computeTaskSchedule(
  now: Date,
  scheduling: TaskScheduling,
  contactTz: string,
  saleTz: string,
  workPattern: WorkPattern,
): { scheduledFor: Date | null; dueAt: Date | null; dueAtAllDay: boolean }
```

1. Si `setScheduledFor === false` → return all null.
2. Compute `targetContactTime` en TZ contact :
   - now + `scheduledOffsetBusinessDays` jours ouvrés (côté contact)
   - heure forcée à `preferredHour`
   - assure que c'est dans `businessHours` et un `allowedWeekdays`
3. Convertit en UTC → c'est `scheduledFor`.
4. Si `setDueAt` → `dueAt = scheduledFor + dueOffsetBusinessDays`
   (jours ouvrés contact, à minuit local sale si `dueAtAllDay`).
5. Borne dure : 14 jours, sinon `NoSchedulableSlotError`.

### Algorithme `findNextFreeSlot` (anti-conflit + quotas + work pattern)

```ts
function findNextFreeSlot(
  desired: Date,
  durationMin: number,
  taskType: TaskType,
  member: { tz; maxEmailsPerDay; maxCallsPerDay; workPattern },
  existingTasks: { scheduledFor: Date; estimatedDurationMinutes: number; type: TaskType }[],
  scheduling: TaskScheduling,
  contactTz: string,
): Date
```

Pour chaque jour candidat à partir de `startOfDay(desired, saleTz)` (max 14 j) :
1. **Quota** : si `count(tasks of type, day) >= max{Type}PerDay` → day+1.
2. **Work pattern windows** ce jour (en saleTz) : list of `(start, end)`.
3. Pour chaque window, scan avec curseur :
   - skip les `existingTasks` qui chevauchent (`+ 5 min buffer`)
   - si on trouve un slot ≥ `durationMin` → retourne
4. Si aucune fenêtre du jour ne marche → day+1.
5. Le contact prime : si la step `businessHours` contact projetée en sale TZ
   tombe **hors** des `work_pattern` windows, on **ignore** les windows
   (hint) et on cherche dans la plage contact en saleTz quand même.

### Wiring engine

`engine-executor-services.createTask` enrichi :
1. Loader fournit : `member.timezone`, `member.workPattern`, quotas,
   `contactTz` cascadé, tâches voisines de l'assignee
2. `{ scheduledFor, dueAt, dueAtAllDay } = computeTaskSchedule(...)`
3. `if (scheduledFor)` → `scheduledFor = findNextFreeSlot(scheduledFor, ...)`
4. Insert avec `scheduledFor`, `dueAt`, `dueAtAllDay`,
   `estimatedDurationMinutes`

## Risques

1. **TZ pourries en base** : select IANA strict côté UI, Zod refuse les
   valeurs hors liste.
2. **work_pattern mal configuré** : windows qui se chevauchent ou ordre
   inversé → Zod refuse, défaut applique 9-12 + 14-17.
3. **Perf slot-finder** : query 14 j max pour un sale chargé → index
   `(assignee_id, due_at)` existe, on ajoute `(assignee_id, scheduled_for)`.
4. **Drift au republish** : la timeline référence `step_id` qui n'existent
   plus (soft-ref par design) → label « Étape supprimée » sans crasher.
5. **Fuseaux extrêmes** : contact prime, sale s'adapte. Si tâche tombe à
   3h du matin sale, c'est comme ça (le sale peut replanifier manuellement).
6. **Page enrolment lourde** : server component, requête unique optimisée.
