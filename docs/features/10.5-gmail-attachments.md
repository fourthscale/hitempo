# 10.5 ‚Äî Gmail outbound attachments (PDF)

> Sprint 10 extension. Adds PDF attachment support to the "Send via Gmail" flow
> initiated from the `GenerateMessageDialog`. Persists files to Supabase Storage
> so they're retrievable from the contact timeline alongside the message.

## Scope

- Up to **4 PDF files per send**, **15 MB each**, **20 MB combined** (safely under the Gmail 25 MB request cap after base64 overhead).
- Files travel as native `File` entries inside the dialog's FormData, server-side
  validated again before being handed to the MIME builder.
- Storage layout : `message-attachments/{organization_id}/{message_id}/{uuid}-{filename}`.
- Download via short-lived signed URLs (5 min TTL) issued on-click from the timeline.
- No file library / template catalog ‚Äî that ships V1.
- No drag-and-drop for MVP (file picker only).

Out of scope :
- HTML body
- Inline images (Content-ID)
- Resend with the original attachments
- Other mime types (DOCX, XLSX, PNG, etc.)

## Architecture

### Strategy : MIME message construction

`lib/gmail/mime-message-strategy.ts` introduces a small Strategy pattern :

- `TextOnlyMimeStrategy` ‚Äî single-part `text/plain` (current MVP behavior)
- `MultipartMixedMimeStrategy` ‚Äî `multipart/mixed` with text body + N file parts (base64, 76-char line-wrapped per RFC 2045)
- `MimeMessageBuilder.forInput(input)` ‚Äî static factory that picks the right strategy based on `input.attachments?.length`

This replaces the prior private `buildMimeMessage` helper inside `gmail-service.ts`. Same call site (`GmailServiceFactory.getInstance().send`), the caller is now MIME-format agnostic.

### Storage : private bucket + RLS-mirrored policies

`message-attachments` is a private Supabase Storage bucket. The first path segment is the org id, which lines up with the storage RLS policies ‚Äî a user can only read/write objects under their own org's prefix. Platform admins keep cross-org read access.

`AttachmentStorageService` is a thin Facade :

- `upload({ organizationId, messageId, filename, mimeType, content })` ‚Üí `{ storageBucket, storagePath }`
- `signedDownloadUrl(bucket, path, ttlSeconds=300)` ‚Üí URL
- `deleteQuietly(bucket, path)` ‚Äî used only by the action's `garbage-collect` path (which today no longer triggers, see below)
- `download(bucket, path)` ‚Üí `Buffer` (reserved for future "resend with attachment" flow)

### Action flow (commit-on-Gmail-send)

`sendMessageViaGmailAction` :

1. Parse + validate scalars (`commitSchema`)
2. Parse + validate `File[]` attachments (`parseAttachmentsFromFormData`) ‚Äî defense in depth on top of the client-side check
3. Gmail send with `attachments` carried in the multipart MIME
4. `persistSentMessage` (new `attachments` arg) :
   1. Insert the `messages` row
   2. Patch `llm_usage` backref
   3. Log the outbound `interactions` row (`status='sent'`, `messageId` set)
   4. For each attachment : `AttachmentStorageService.upload()` ‚Üí `insertMessageAttachment()`. Failures here are logged but NOT thrown ‚Äî Gmail already sent the message, the user can re-upload later if it matters
   5. `completeTask` if `taskId` present

**Why upload to Storage AFTER Gmail succeeds**, not before : it removes the "garbage-collect orphaned blobs on Gmail failure" path. The price is degraded UX when Storage fails after Gmail (message sent, no archive) ‚Äî acceptable for the dogfood phase.

### DB schema

`message_attachments` (`db/schema.ts` + migration `20260528222725_message_attachments.sql`) :

| column | type | notes |
|---|---|---|
| `id` | `uuid PK` | |
| `organization_id` | `uuid FK ‚Üí organizations` | CASCADE |
| `message_id` | `uuid FK ‚Üí messages` | CASCADE ‚Äî drop message ‚áí drop attachments |
| `storage_bucket` | `text` | default `'message-attachments'`, not hardcoded in code |
| `storage_path` | `text` | `{orgId}/{msgId}/{uuid}-{filename}` |

Indexes : `idx_message_attachments_message`, `idx_message_attachments_org`.

RLS : `SELECT` (org members OR platform admin), `INSERT/UPDATE/DELETE` (org members only).

### UI

- `AttachmentsPicker` lives at the bottom of `GenerateMessageDialog.tsx`, rendered only when `channel === "email" && gmail.connected`.
- Hidden `<input type="file" accept=".pdf,application/pdf" multiple>` triggered by a labelled button.
- Per-file row with filename, size, remove icon.
- Inline validation errors (i18n'd) shown below the picker.
- On Gmail-send : files are appended to FormData under the key `attachments`.

- `InteractionsTimeline` renders an `AttachmentsList` below the summary of any interaction whose `messageId` has attachment rows. Click ‚Üí server action `getAttachmentDownloadUrlAction` issues a fresh signed URL ‚Üí `window.open(url, "_blank", "noopener")`.
- Contact page bulk-fetches attachments for all timeline messages via `getAttachmentsByMessageIds`.

## Acceptance criteria

- [ ] Add 1 PDF (~500 KB), send via Gmail ‚Üí email arrives in recipient inbox with the PDF attached.
- [ ] After send, the contact's timeline lists the PDF below the outbound interaction row.
- [ ] Click on the PDF in the timeline opens it in a new tab.
- [ ] Try to upload a non-PDF ‚Üí inline error, file rejected client-side, no server call made.
- [ ] Try to upload 5 PDFs ‚Üí 4th is accepted, 5th rejected with "Maximum 4 attachments‚Ä¶".
- [ ] Try to upload a 20 MB PDF ‚Üí rejected with "exceeds 15 MB per-file cap".
- [ ] Multi-tenant : open the timeline as a different org ‚Üí no attachments leak.
- [ ] Storage failure after Gmail success ‚Üí message + interaction still appear in the UI, attachment list is empty (server log shows the failure).

## Implementation notes

- The `AttachmentRejectedError` thrown by the server action is surfaced inline in the dialog (existing `gmailError` state path) rather than via the global `ActionErrorModal`. Less interruptive for a recoverable error.
- `AttachmentNotFoundError` (from the download action) bubbles up to the timeline component and shows a small inline error ‚Äî no modal.
- The `lib/gmail/attachment-limits.ts` constants are imported by both client and server. The file has no `"server-only"` directive on purpose.
- The MIME builder Strategy is genuinely useful here, not OOP-for-OOP : text-only and multipart messages have completely different byte layouts, and an HtmlMimeStrategy / SignedMimeStrategy can slot in later without touching `GmailService`.
- **Next.js 16 has TWO body-size caps in the request path**, both default to 10 MB and BOTH need to be raised for a PDF send to work :
  - `experimental.proxyClientMaxBodySize` ‚Äî the middleware / proxy layer cap. The Supabase auth middleware runs on every request, so this is the FIRST gate. Raised to 25 MB.
  - `experimental.serverActions.bodySizeLimit` ‚Äî the Server Action body cap (defaults 1 MB, not 10). Also raised to 25 MB.
  - Both sit above our 20 MB combined-attachments cap to leave room for form fields + multipart framing. Bump both in sync if `attachment-limits.ts` changes.
  - The error message points to `middlewareClientMaxBodySize` (the deprecated name) but only the new `proxyClientMaxBodySize` is in the public type.
