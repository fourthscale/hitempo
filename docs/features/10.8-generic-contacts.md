# 10.8 — Generic contacts

> Small pre-sprint to unblock Sprint 11 (sequences). Allows creating a
> contact when only a generic channel is known (`info@hotelwestminster.com`
> or a switchboard phone) — no need to invent a first/last name. The
> contact still exists in the database so all the prospection machinery
> (interactions, messages, lastContactedAt, sequences) keeps flowing
> through a single entity model.

## Why now

- L&G dogfood reality : the website / front desk often only exposes a
  generic email and a switchboard number. The current schema forces
  `first_name` and `last_name` to be `NOT NULL`, so reps invent placeholder
  values ("Contact" / "Westminster") which pollutes UI lists, search, and
  the AI prompt builder.
- Sprint 11 (sequences) ships the auto-enrollment story. Without generic
  contacts, the first sequence enrolment on a new hotel would either fail
  or carry junk data. Better to fix the substrate before piling on.

## Scope

In :
- New `contact_kind` enum : `'person' | 'generic'`.
- `contacts.first_name` and `contacts.last_name` become `NULL`-able.
- Application-level invariants enforced via Zod :
  - `kind = 'person'`  → `first_name + last_name` required, `email`/`phone` optional.
  - `kind = 'generic'` → `first_name + last_name` may be null, but at least
    one of `email` / `phone` required.
- Display name resolver (`resolveContactDisplayName`) used everywhere a
  contact name is rendered :
  - `kind = 'person'`  → `"{first} {last}"` (or just one if the other is null).
  - `kind = 'generic'` → `"Contact générique · {company.name}"`.
- AI prompt builder fallback for generic contacts : no `{firstName}` salutation,
  uses a neutral `"Bonjour"` / `"Hello"` opener, and the contact role (if set)
  is referenced functionally instead of by name.
- Sender / recipient header in the Gmail send : the recipient name part is
  dropped for generic contacts (just the bare email), avoiding `"Contact <info@…>"`.
- Forms + CSV import recognize a `kind` column / radio. Default = `'person'`
  for backwards compatibility on rows that omit it.

Out of scope :
- Promoting a generic contact to a person once a real name is learned :
  that's just an update of `kind` + names ; no migration story needed for MVP.
- Adding `email`/`phone` directly on `companies` / `sites` : the contact stays
  the canonical entity ; keeping interactions, messages, optedOut, etc.
  keyed on one shape is worth one extra row per company.

## Data model

```
contacts
  ALTER COLUMN first_name DROP NOT NULL
  ALTER COLUMN last_name  DROP NOT NULL
  ADD COLUMN kind contact_kind NOT NULL DEFAULT 'person'

new enum: contact_kind = ('person' | 'generic')

-- DB-level CHECK constraint (Phase A defense in depth, application Zod
-- is the primary guard):
ADD CONSTRAINT contacts_kind_consistency CHECK (
  (kind = 'person'  AND first_name IS NOT NULL AND last_name IS NOT NULL) OR
  (kind = 'generic' AND (email IS NOT NULL OR phone IS NOT NULL))
)
```

Migration is purely additive : existing rows default to `'person'`, none
of them violate the constraint because `first_name`/`last_name` were
already `NOT NULL` before.

The `full_name` generated column (already in DB) keeps the existing
COALESCE-ish formula — for generic contacts it will be `null`, callers
should not rely on it. Display logic uses `resolveContactDisplayName`
instead, which centralises the fallback.

## Implementation plan

### Slice 1 — Schema + migration
- Add `contact_kind` pgEnum + `kind` column in `db/schema.ts`.
- Drop `notNull` on `first_name` / `last_name`.
- Generate Drizzle migration, append the CHECK constraint manually, sync
  to `supabase/migrations/`, apply to local + cloud Supabase.

### Slice 2 — Application invariants + display helper
- New `lib/contacts/contact-kind.ts` with the Zod schemas (`personContactSchema`,
  `genericContactSchema`, discriminated union) and `resolveContactDisplayName(c)`.
- Wire the schemas into `createContactAction` / `updateContactAction`.
- Update `lib/messages/contact-display.ts` (or wherever names are formatted)
  to use the resolver.

### Slice 3 — UI
- Contact create / edit forms : radio "Type de contact" : Personne / Générique.
  Hide first/last when Générique is selected, show a hint about email/phone
  being required.
- Contact lists + cards + headers : use the resolver everywhere
  (`{firstName} {lastName}` → `resolveContactDisplayName(contact)`).
- CSV import : `Contacts` template gains a `kind` column (defaults to
  `person`) — backwards compatible (omitted column → person).

### Slice 4 — AI prompt + Gmail send
- `buildOutboundMessagePrompt` : when contact is generic, swap the
  personalised salutation block for a neutral one (`Bonjour,` / `Hello,`)
  and reference the contact functionally (e.g. "À l'équipe X" / "To the
  team at X").
- Gmail send recipient header : strip the display-name part for generic
  contacts, send just the email.

### Slice 5 — Verify
- tsc + lint + vitest + production build.
- Manual : create a generic contact via UI, send it a message via Gmail,
  verify the rendered subject + body have no "Bonjour {undefined}".
- Manual : check that a "person" contact still flows exactly as before
  (no regression).

## Acceptance criteria

- [ ] A contact can be created with `kind = 'generic'`, no first/last name,
      just an email.
- [ ] DB rejects an INSERT of a generic contact with no email and no phone.
- [ ] UI lists show `"Contact générique · {company}"` for generic contacts.
- [ ] AI prompt for a generic contact does not produce a `Bonjour undefined`
      / `Hello {firstName}` artefact.
- [ ] Gmail send To header for a generic contact is `<info@hotel.fr>`
      not `"undefined undefined" <info@hotel.fr>`.
- [ ] CSV import recognises a `kind` column ; omitting it defaults to person.
- [ ] All existing tests pass, no regression on person contacts.

## Implementation notes

Shipped as planned. Key decisions + deviations :

- **Display resolver is locale-agnostic** (`lib/contacts/contact-kind.ts`
  → `resolveContactDisplayName`). For a generic contact it returns the
  email (or phone), which reads naturally as "this is a channel, not a
  person" without needing translation. The brief's "Contact générique ·
  {company}" idea was dropped in favour of showing the actual email —
  more informative, and avoids threading i18n + the company name through
  ~18 call sites. A `genericFallback` option exists for the (never-hit
  given the CHECK) case where both email and phone are null.

- **Nullable names cascaded widely.** Making `first_name`/`last_name`
  nullable surfaced ~9 tsc errors across pages/components that assumed
  `string`. Fixed by (a) widening the consumer types to `string | null`
  + `kind`, and (b) routing every contact-name render through the
  resolver. Restricted contact column selections in `db/queries/*`
  (tasks, interactions, companies) were extended with `kind` + `email`
  so the resolver works everywhere. The annotator's term filter already
  drops empty terms, so a generic contact's null name simply isn't
  highlighted.

- **AI prompt** (`outbound-message-prompt.ts`) gained a `kind` field on
  its `contact` input. Generic contacts get a system instruction to use
  a neutral salutation ("Bonjour," / "Hello,") and never invent a name,
  plus the `## Contact` line shows "Adresse générique" instead of a name.

- **Gmail recipient header was already clean** — `sendMessageViaGmailAction`
  passes `to: contact.email` and the MIME builder emits `To: <email>`
  with no display-name part, so there was never an "undefined undefined"
  risk. Slice 4 reduced to the AI-prompt change.

- **CSV import — BOTH modes** handle generic contacts :
  - `contacts` mode : `kind` column (defaults `person`), nullable
    names, `superRefine` enforcing the per-kind invariant.
  - `all-in-one` mode : `contact_kind` column (defaults `person`).
    Section detection updated — a contact section now exists when a name
    is present OR `contact_kind=generic` with a channel. The names-required
    check became kind-aware (`validateContactSection`). This matters
    because all-in-one is the prime path for importing a hotel + site +
    its `info@` contact in a single row.
  - Both downloadable templates (header + example) gained the kind
    column. The `/api/import/template` route renders straight from the
    `csv-template-headers.ts` spec, so the columns propagate
    automatically — no hardcoded header list anywhere in the UI or route.

- **DB CHECK constraint** `contacts_kind_consistency` is the defense-in-
  depth backstop ; Zod at the action + importer layers is the primary
  guard. Migration is purely additive — existing rows default to
  `person` and already had non-null names.

- **Tests** : `tests/contacts/contact-kind.test.ts` covers the resolver
  fallback chain + the Zod invariants (14 cases). Pre-existing failures
  in `message-generation-orchestrator.test.ts` (4) and the `rls/*`
  suites are unrelated to this sprint (they fail on `main` too —
  verified via git stash).

- **Generic contacts hide person-only fields in the form.** A generic
  address (info@, switchboard) has no first/last name, no job title, no
  personal LinkedIn, and the person-centric `contactRole` enum
  (decision_maker / influencer / …) doesn't apply. So `ContactKindFields`
  now owns all of those — firstName, lastName, jobTitle, role, linkedinUrl
  — and renders them only in the `person` branch. The generic branch shows
  just the kind toggle + a hint. The columns stay in the DB (nullable) ;
  switching a contact person→generic on edit clears them (not submitted →
  set to null), which is the correct semantics. We deliberately did NOT
  add a "generic"/"contact" value to the `contactRole` enum — it would be
  redundant with `kind`. If categorising generic inboxes later
  (reservations@, rh@, info@) is wanted, that's a separate "inbox
  function" concept, not the person-role enum.

- Migration applied to **cloud + local** Supabase.
