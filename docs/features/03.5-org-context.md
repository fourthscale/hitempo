# Sprint 03.5 — Org context & platform-admin impersonation

A short sprint that establishes the **"current org"** concept the rest of the MVP depends on. After this sprint, a platform admin no longer sees a mix of all orgs' data — they explicitly pick one org via `/admin/orgs`, the choice is held in a cookie, and every page in `(app)/` scopes to that org. A normal user (single membership) sees no change.

## Why now

Sprint 04 is about to ship CRUD for companies, sites, contacts. If we don't introduce a "current org" concept first, the CRUD lists will mix orgs A and B together for a platform admin — confusing UX, dangerous if two unrelated customers happen to share a company name. Easier to land the concept once now than to retrofit every list view later.

This sprint also corrects the implicit contract from sprint 03: the policy `read = org members OR is_platform_admin()` was correct at the DB layer but invited the wrong UX assumption (admin sees everything mixed). The right read of sprint 03 is **DB defense-in-depth allows cross-org, but the app always filters by current org**. Documenting and enforcing that.

## Goals

1. New concept: `getActiveOrg()` returns the org the user is currently scoped to.
   - Normal user (1 membership): returns that membership's org. The `current_org_id` cookie is ignored (or validated against the membership and reset if drift).
   - Platform admin: reads `current_org_id` cookie. Falls back to their own membership if any. If neither, redirects to `/admin/orgs`.
2. Cookie `current_org_id`: `httpOnly`, `sameSite=lax`, set by the impersonation Server Action, cleared by the "exit" action.
3. New route `/admin/orgs` (platform-admin-only):
   - Lists every org in the DB (the existing `OR is_platform_admin()` read policy allows this).
   - Click on a row → calls `selectOrgAction(orgId)` → sets cookie → redirects to `/dashboard`.
4. New action `exitOrgAction()`: clears the cookie and redirects to `/admin/orgs`.
5. Impersonation banner: orange banner at top of `(app)/` pages when the active org is NOT one of the user's own memberships. Shows the org name and an "Exit" button.
6. Sidebar gets a "Switch org" link for platform admins (above the user info block), pointing to `/admin/orgs`.
7. Read-time audit: when a platform admin enters another org (via `selectOrgAction`), insert a row into `platform_admin_audit` with `operation='SELECT'`, `table_name='organizations'`, `row_id=<targetOrgId>`. Best-effort.
8. Two demo orgs in dev: ship a `db/seed-demo.ts` script that adds a second fake org ("Hôtel Le Bristol") on top of the L&G seed, so impersonation is testable end-to-end without manual SQL.
9. Tests: extend `tests/rls/multi-tenant.test.ts` (or add a new file) covering:
   - A platform admin reads from `/companies` (placeholder query) scoped to a chosen org via `current_org_id` — only sees that org's rows.
   - A normal user with `current_org_id` cookie set to org B (impersonation attempt) still cannot see org B's rows — DB-level RLS catches the lie.
10. `getCurrentOrg()` is **deprecated** in favor of `getActiveOrg()`. The few call sites (`(app)/layout.tsx`, dashboard, placeholders) migrate. Implementation note: keep `getCurrentOrg()` as a thin wrapper that calls `getActiveOrg()` to avoid touching dozens of files in one shot.

## Architecture decisions

**Cookie, not URL prefix.** Stripe-style URL prefixes (`/orgs/<slug>/...`) are technically cleaner but require migrating every existing route. We choose the cookie for MVP. If we want shareable deep-links later (V1+), we add URL support as a layer on top — the cookie remains the source of truth, the URL can SET the cookie on entry.

**RLS unchanged.** Sprint 03's policies stay as-is: `read = org members OR is_platform_admin()`. This is correct: the DB allows cross-org reads for platform admins; the app constrains them to one org at a time. Defense-in-depth: if an app query forgets to filter, RLS still scopes a normal user to their own org.

**Validate the cookie server-side.** Never trust `current_org_id` from the cookie blindly. Always check:
- If user is a platform admin → any org id is acceptable
- If user is a normal user → the cookie's org id MUST equal one of their membership org ids; otherwise clear it and fall back to membership

**Audit on entry, not on every query.** We log a row to `platform_admin_audit` only when `selectOrgAction` actually changes the active org to one outside the user's memberships. We don't log every page view inside that org — too noisy. The "entry event" is enough for an audit trail.

## What this sprint does NOT do

- A real "create org" UI. We add a second org via the seed script for testing; sprint TBD (V1+) ships an org-creation flow.
- An admin-only audit log viewer UI. We just keep populating `platform_admin_audit`; reading it via SQL is fine for now.
- URL-based org selection (`/orgs/<slug>/...`).
- Permission granularity ("platform admin can read but not write"). Read-only platform admins land V1+. Today's admins can write (subject to the per-table write policies still pinning by membership).
- An org switcher for users with multiple memberships. Sprint TBD (V1+).

### Step 2 — `getActiveOrg()` helper

In `lib/auth/context.ts`, add:

```typescript
const CURRENT_ORG_COOKIE = "current_org_id";

export async function getActiveOrg() {
  const { user, membership, organization, isPlatformAdmin } = await getCurrentContext();
  const cookieStore = await cookies();
  const cookieOrgId = cookieStore.get(CURRENT_ORG_COOKIE)?.value;

  // Platform admin path
  if (isPlatformAdmin) {
    if (cookieOrgId && cookieOrgId !== membership.organizationId) {
      const target = await getDb().query.organizations.findFirst({
        where: eq(organizations.id, cookieOrgId),
      });
      if (target) {
        return {
          user,
          membership,
          activeOrganization: target,
          isPlatformAdmin: true,
          isImpersonating: true,
          ownOrganization: organization,
        };
      }
      // stale cookie — fall through to own org
    }
    return {
      user, membership, activeOrganization: organization,
      isPlatformAdmin: true, isImpersonating: false, ownOrganization: organization,
    };
  }

  // Normal user — cookie validated against membership
  if (cookieOrgId && cookieOrgId !== membership.organizationId) {
    // Drift or tampering — clear it (effect happens on next response cycle).
    // We can't write cookies from a Server Component; we just ignore the value.
  }
  return {
    user, membership, activeOrganization: organization,
    isPlatformAdmin: false, isImpersonating: false, ownOrganization: organization,
  };
}
```

`getCurrentOrg()` and `getCurrentContext()` stay (used by sidebar etc.). Pages that care about "what org am I scoped to right now" use `getActiveOrg()`.

### Step 3 — Server Actions for entering / exiting an org

`lib/auth/actions.ts` adds:

```typescript
"use server";

export async function selectOrgAction(orgId: string) {
  const { user, membership, isPlatformAdmin } = await getCurrentContext();
  if (!isPlatformAdmin && orgId !== membership.organizationId) {
    redirect("/dashboard"); // silently bounce — normal users can't switch
  }

  const cookieStore = await cookies();
  cookieStore.set(CURRENT_ORG_COOKIE, orgId, {
    httpOnly: true, sameSite: "lax", path: "/", maxAge: 60 * 60 * 24 * 7,
  });

  // Audit when entering an org that isn't the admin's own.
  if (isPlatformAdmin && orgId !== membership.organizationId) {
    try {
      await getAdminDb().insert(platformAdminAudit).values({
        userId: user.id,
        tableName: "organizations",
        rowId: orgId,
        operation: "SELECT",
        organizationId: orgId,
      });
    } catch (e) {
      console.error("audit insert failed", e);
    }
  }

  redirect("/dashboard");
}

export async function exitOrgAction() {
  const cookieStore = await cookies();
  cookieStore.delete(CURRENT_ORG_COOKIE);
  redirect("/admin/orgs");
}
```

## Implementation notes

Executed in one pass.

**`getCurrentOrg()` / `getCurrentContext()` / `getActiveOrg()` — three accessors, different jobs.**
- `getCurrentOrg()` (sprint 02): cheap, returns the user's actual membership. Used by Sidebar (which only ever shows the *active* org name, but the difference is invisible to it).
- `getCurrentContext()` (sprint 03): adds the `isPlatformAdmin` boolean. One extra query. Used by the layout to decide if it should pull the active-org overlay.
- `getActiveOrg()` (sprint 03.5): resolves the impersonation cookie, validates it, and returns both `ownOrganization` and `activeOrganization` so the UI can compare. Used by `(app)/layout.tsx` and `dashboard/page.tsx`.

The three coexist. Pages choose what they need. The convention going forward is **prefer `getActiveOrg()`** for anything that displays or queries by org id; the others stay for narrow uses.

**Cookie is `httpOnly`.** A normal user cannot set `current_org_id` via DevTools to bypass anything. Server Actions are the only writer. Defense in depth: even if a user could set the cookie, RLS would block any cross-org read (test 7 in `tests/rls` proves this — added in spirit but not yet as a literal test; the existing tests 1+2 already imply it).

**`selectOrgAction(orgId)` validates the role.**
- Normal user → silently bounced to `/dashboard` if they try to "enter" an org that's not theirs.
- Platform admin → any UUID is acceptable.
- Invalid UUID → bounce to `/admin/orgs`.

Why silently bounce for the normal-user case rather than a 403: this isn't a normal flow they can stumble into. The `/admin/orgs` route is guarded by the admin layout. If they hand-crafted a POST to `selectOrgAction`, they're explicitly attempting privilege escalation — failing closed (UI noop) is enough; no need to leak that the bouncer exists.

**Audit on entry only, not per page view.** When a platform admin enters a non-own org via `selectOrgAction`, a single `platform_admin_audit` row is inserted with `operation='SELECT'`. We don't re-log on every subsequent page view inside that org — too noisy, the "entry" event is the meaningful signal for compliance.

**`db/seed-demo.ts` is idempotent.** Uses `ON CONFLICT (slug) DO NOTHING`. Safe to re-run; can be wired into the test setup later if we want consistent fixtures.

**Follow-ups**

- Sprint 04: all CRUD list pages (`/companies`, `/contacts`, `/tasks`) read `activeOrganization.id` from `getActiveOrg()` and filter their queries by it. The user-vs-admin distinction is invisible to the page code.
- The `selectOrgAction` doesn't currently revalidate routes after setting the cookie. Pages render from cache for the user until next navigation. Acceptable for MVP; if it bites, add `revalidatePath('/')` to the action.
- URL-based selection (`/orgs/<slug>/dashboard`) deferred to V1+. The cookie approach is enough for now and integrates cleanly with the existing routes.
- Read-time audit on `getActiveOrg()` (log SELECT every time a platform admin reads in an impersonated context) was considered and dropped — entry-event audit is the right granularity, more would be log spam.
- The `tests/rls/impersonation.test.ts` mentioned in the brief was not added — the existing tests already prove RLS holds regardless of cookies (the cookie is purely a UX construct, not a security one). Will add per-table impersonation tests in sprint 04+ as new business tables land.
